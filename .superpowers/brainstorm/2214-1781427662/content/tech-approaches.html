三种技术方案对比

从个人项目的角度出发，重点评估：开发效率、3D能力、UI集成、后续扩展

基础技术共识

无论选哪个方案，以下层面是通用的：

🗺 Three.js — 3D渲染引擎
📦 GLTF/GLB — 3D模型格式
🗄 IndexedDB / JSON — 本地数据存储
📐 Vite — 构建工具
🔤 TypeScript — 类型安全
A

React + React Three Fiber ⭐ 推荐

React 驱动 UI，R3F 声明式管理 3D 场景。组件化：一个 Panel 组件 = 一个 .tsx 文件。状态管理用 Zustand。UI 与 3D 紧密耦合的场景最适合。

✅ 组件化开发，3D/UI 复用同一生态
✅ 大量 Three.js 封装，代码量少 40%
✅ 摄像机动画用 @react-three/drei
✅ 个人项目社区资源最丰富
⚠️ 学习 R3F 声明式思维
⚠️ React 体积 ~140KB
B

Vanilla Three.js + 轻量框架

直接使用 Three.js 管理 3D 场景，UI 层用 Svelte 或 Preact。3D 场景完全手动控制，适合需要精细优化 3D 性能的场景。

✅ 完全控制 3D 渲染管线
✅ 包体积最小
✅ Svelte 编译后极轻量
⚠️ 3D/UI 通信需要手动桥接
⚠️ 自定义摄像机动画代码量大
⚠️ 缺少声明式 3D 组件封装
C

Electron + React + R3F（桌面应用）

与 A 相同技术栈，但打包为桌面应用。可访问文件系统（直接导入本地模型）、提供原生窗口体验。

✅ 文件系统访问（本地模型导入）
✅ 原生体验，无浏览器限制
✅ 可离线使用
⚠️ 打包体积大（~150MB）
⚠️ 更新需要重新打包
⚠️ 超出 MVP 需求

推荐方案 A：React + R3F 的架构蓝图

🧱 核心依赖
• React 18 + TypeScript
• Three.js + @react-three/fiber
• @react-three/drei（摄像机、OrbitControls、辅助工具）
• Zustand（全局状态管理：地形数据、UI状态）
• Vite（构建）
• React Router（页面路由：地图/人物/体系/时间线）
📁 项目结构示意
src/
├── components/
│ ├── scene/ # 3D 场景（WorldMap, Terrain, Camera）
│ ├── editor/ # 编辑器（GridEditor, ModelPalette）
│ ├── panels/ # 信息面板（DetailPanel, PersonList）
│ └── ui/ # 通用 UI（ToggleSwitch, Toolbar）
├── pages/ # 路由页面
├── store/ # Zustand stores
├── data/ # 地形定义、模型配置
└── types/ # TypeScript 类型定义

后续扩展能力对比

方案 A (R3F) 🟢
→ AI 接入：直接调用 API
→ Obsidian 插件：组件复用
→ 人物关系图谱：React 图谱库集成
→ 时间轴：同一路由体系
方案 B (Vanilla) 🟡
→ AI 接入：可
→ 插件化：需自行封装 API
→ 图谱集成：可以，但桥接代码多
方案 C (Electron) 🔵
→ 最强本地能力
→ 但 MVP 阶段过度设计
→ Web 版本后续可升级至 Tauri
