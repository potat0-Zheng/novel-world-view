# 小说世界观可视化应用 — 项目规划书 v0.1

> 本文档用于记录项目需求、设计方案与技术选型，供交叉审阅与后续开发参考。

---

## 一、项目概述

### 1.1 项目目标

创建一个以网络小说类型为主的世界观可视化应用。用户可以在与应用交互的过程中，自定义搭建自己的小说世界。

### 1.2 核心子系统

世界观搭建分为四个相互关联的子系统：

| 子系统 | 核心形态 | 数据关联 |
|--------|----------|----------|
| 🌍 世界样貌 | 2.5D/2D 交互式地图 | 地理框架下承载人物、事件、势力的位置信息 |
| 👥 人物关系 | 家族树 / 网络图谱 | 人物绑定至地理位置，可筛选时间 |
| 📚 体系与元素 | 可视化列表 | 体系标签可关联至地图位置与人物 |
| ⏳ 时间线 | 线性节点可视化 | 事件绑定至地理位置，人物在事件中出现 |

**核心原则**：所有子系统的数据以「地理位置」为框架相互关联。例如在世界样貌中点击一座城市，可看到该城市中的人物、发生的事件、所属势力与体系。

### 1.3 呈现形态

- **首发形态**：Web 应用（网页端）
- **后续扩展**：插件 / 跳转接入其他软件、接入 AI 辅助操作

## 二、世界样貌 — 核心设计（已完成讨论）

> 世界样貌是本应用的核心模块，以下为该模块的完整设计。

### 2.1 渲染方案

| 项目 | 方案 | 说明 |
|------|------|------|
| **主视角** | 2.5D 等距视角 | 摄像机位于侧上方，倾角约 60°。类似《模拟城市》的视角体验 |
| **备选视角** | 2D 俯视平面 | 一键切换，用于快速定位和全局浏览 |
| **视角切换** | 连续摄像机动画 | 从倾斜 ~60° 平滑过渡至 0°（正上方），0.8s 动画，可来回切换 |
| **模型支持** | 3D 模型（建筑/自然景观） | 仅在 2.5D 视角下渲染，2D 模式下退化为图标+文字标签 |
| **交叉过渡** | 淡入/淡出切换 | 视角切换时，在倾角降至约 15°–20° 开始淡入图标层、淡出模型层，形成交叉过渡，而非在 0° 处突变 |

### 2.2 地图编辑方式

用户通过**网格系统**创建世界：

1. **网格基础**：初始提供 N×N 网格（如 10×10），用户通过选取连续格子圈定陆地/海洋初始板块
2. **数据 vs 视觉**：数据层始终基于单元格（cell-based），但渲染层通过过渡融合算法消除格线，呈现有机的不规则大陆形态
3. **单元格渲染方案**：采用「过渡融合法」——相邻同类地形边界自动柔化，不同类地形之间（如陆地-海洋）自动生成渐变过渡带（浅滩/沙滩）

### 2.3 三层图层架构

数据分为三个独立图层，逐层依赖：

```
L1: 基础地形 ───→ 覆盖全部网格，海洋/大陆互斥，必须完成 L1 才能进入 L2
     ↓
L2: 自然地形 ───→ 仅可放置在 L1「大陆」格上，互斥（一格一种 L2 类型）
     ↓
L3: 人造建筑 ───→ 必须建立在 L1 或 L2 之上，可多个叠加
```

#### L1 基础地形

| 类型 | 说明 | 互斥 |
|------|------|------|
| 🟦 海洋 | 覆盖未选为大陆的格子 | ✅ |
| 🟩 大陆 | 用户圈选为陆地的格子 | ✅ |

#### L2 自然地形（互斥，仅大陆格可选）

| 类型 | 说明 | 优先级 |
|------|------|--------|
| 🌿 平原 | 默认大陆地形 | MVP |
| ⛰ 山脉 | 高海拔，不可通行 | MVP |
| 🌲 森林 | 密林区域 | MVP |
| 🏜 沙漠 | 干旱区域 | 后续 |
| 🏔 雪原/冻土 | 极寒地带 | 后续 |
| 🌋 火山 | 特殊地貌 | 后续 |
| 🌫 沼泽 | 湿地/毒雾区 | 后续 |
| 🪵 丘陵 | 平原↔山脉过渡带 | 后续 |
| 🌊 河流 | 沿网格边线放置，非面状覆盖 | MVP（特殊处理） |

> **扩展性**：每个 L2 类型是一个配置对象（名称、颜色、海拔、是否可行走），新增只需添加一行配置，无需改动渲染逻辑。

#### L3 建筑模型库

> 模型来源：免费 CC0 模型导入（GLTF/GLB）+ AI 生成（后续迭代）
> 风格建议：Low-Poly（低多边形）—— 个人项目开发首选，AI 生成友好

**Demo 模型选取**（每个分类选经典代表）：

| 分类 | 测试模型 | 说明 |
|------|----------|------|
| 🏰 城堡/要塞 | 中型城堡 | 测试建筑在平原上的表现 |
| 🏘 聚落 | 小村庄 | 测试多格分布 |
| ⛩ 殿堂/宗教 | 神殿 | 仙侠/奇幻通用 |
| 🗼 塔/碑 | 法师塔 | 高耸结构在 2.5D 中的表现 |
| 🏗 工事/设施 | 桥梁 | 跨河流边线放置，特殊放置逻辑 |
| 🌳 自然地标 | 巨树/神木 | 点状地标与 L2 森林面的对比 |

### 2.4 交互流程

#### 2.5D 视图交互

1. 用户点击场景中的模型/位置
2. 该位置的**模型弹出、放大、连贯地向页面左侧移动**（原地保留标记）
3. 右侧滑出信息面板，展示该地理单元中包含的信息：
   - 📝 位置描述
   - 👥 关联人物列表
   - 📅 关联事件列表（含时间）
   - ⚔️ 所属势力 / 体系标签

#### 2D 视图交互

用户点击位置 → 弹出全屏/浮层详情列表，信息结构与 2.5D 面板一致，布局更适合平面阅读。

## 三、交互式数据关联

各子系统之间的数据关联方式（以地图位置为锚点）：

```
         ┌──────────────────┐
         │   🌍 地理位置     │ ← 地图上的一个单元/建筑
         └────────┬─────────┘
                  │
          ┌───────┼───────┐
          ▼       ▼       ▼
     ┌──────┐ ┌──────┐ ┌──────┐
     │ 👥 人物 │ │ 📅 事件 │ │ ⚔️ 势力 │
     │ 位置锚定  │ │ 时间锚定  │ │ 体系归属  │
     │ 时间筛选  │ │ 人物参与  │ │ 位置绑定  │
     └──────┘ └──────┘ └──────┘
```

后续三个子系统（人物关系、体系与元素、时间线）作为独立的页面/视图存在，但其数据模型以地理位置为交叉索引。

### 3.1 数据引擎设计

四个子系统以「地理位置」为交叉索引，构成一个轻量图数据库的语义：

```
位置 ──→ 人物（n:n，含时间约束）
位置 ──→ 事件（n:n，含时间锚定）
位置 ──→ 势力（n:n）
人物 ──→ 事件（参与关系）
```

**核心原则：单一数据源 + 索引模式**。不允许多个子系统各自维护一份数据副本，所有数据集中在单一源中，通过空间索引和时间索引实现交叉查询：

```typescript
interface WorldData {
  entities: Map<Id, Entity>;         // 唯一数据源
  locationIndex: Map<LocationId, Set<EntityId>>;  // 空间索引
  timeIndex: Map<Timestamp, Set<EntityId>>;       // 时间索引
}
```

这个数据架构应在 **Phase 1** 就确定并搭建（即使 MVP 只用到其中一部分），以避免 Phase 2 做交叉引用时出现数据同步问题。

### 3.2 序列化策略

同时支持两种存储方式，在 Phase 1 建立：

| 方式 | 适用场景 | 实现要求 |
|------|----------|----------|
| **IndexedDB** | 自动持久化（类似 Figma 的自动保存）、大体积数据、离线使用 | 每次变更后自动写入，用户无感知 |
| **JSON 导入/导出** | 用户主动分享、备份、版本控制、跨设备传输 | 使用文件选择器 API，一次性全量导入/导出 |

两者应共用同一套序列化/反序列化逻辑，确保数据格式一致。IndexedDB 是默认存储后端，JSON 导入/导出是用户层的传输格式。

## 四、技术方案选择

### 4.1 技术共识（无论选哪个方案）

| 层面 | 技术 |
|------|------|
| 3D 渲染引擎 | Three.js（事实标准） |
| 3D 模型格式 | GLTF / GLB |
| 本地数据存储 | IndexedDB / JSON 文件 |
| 构建工具 | Vite |
| 编程语言 | TypeScript |

### 4.2 三种方案对比

#### 方案 A：React + React Three Fiber（R3F）⭐ 推荐

| 项目 | 说明 |
|------|------|
| **核心栈** | React 18 + Three.js + @react-three/fiber + @react-three/drei |
| **状态管理** | Zustand（地形数据、UI 状态） |
| **路由** | React Router（4 个子系统页面） |
| **优点** | ① 声明式 3D 场景管理，代码量少 40%；② UI 与 3D 在同一组件生态内无缝集成；③ @react-three/drei 提供现成的摄像机动画与控制器；④ 个人项目社区资源最丰富 |
| **缺点** | 需要学习 R3F 声明式思维；React 基础体积 ~140KB |
| **扩展性** | AI 接入 → 直接调用 API；谱图集成 → React 图谱库（如 visx、d3）；插件化 → Web Component / iframe 嵌入 |

#### 方案 B：Vanilla Three.js + 轻量框架（Svelte / Preact）

| 项目 | 说明 |
|------|------|
| **核心栈** | Three.js + Svelte / Preact |
| **优点** | 完全控制 3D 渲染管线；包体积最小 |
| **缺点** | 3D 与 UI 通信需手动桥接；摄像机动画代码量较大；缺少声明式 3D 组件封装 |
| **扩展性** | 可行但桥接代码较多 |

#### 方案 C：Electron + React + R3F（桌面应用）

| 项目 | 说明 |
|------|------|
| **核心栈** | 同方案 A + Electron 打包 |
| **优点** | 文件系统访问（本地模型导入）；原生窗口体验；可离线使用 |
| **缺点** | 打包体积大（~150MB）；更新需重新打包；**MVP 阶段过度设计** |

### 4.3 推荐结论

**推荐方案 A（React + R3F）**。理由：
1. 本应用是**UI 密集型**应用（大量面板、列表、编辑器），React 组件化天然适合
2. 2.5D 场景的摄像机控制、模型管理通过 R3F 声明式编写，比原生 Three.js 减少大量胶水代码
3. 后续三个子系统（人物关系图谱、体系列表、时间线）直接复用 React 生态中的图谱/可视化库
4. 从 Web 到桌面（Tauri/Electron）或插件化的迁移路径清晰

> **远期备选：Svelte + Threlte**。Threlte 是 Svelte 生态中的 R3F 等价物，提供类似的声明式 3D 体验，同时享有 Svelte 的编译时优化（更小的包体积）。目前不推荐替换方案 A，因为 Threlte 社区规模远小于 R3F，个人项目遇到问题更难找到答案。但值得在未来关注——如果某天对包体积有极致追求，迁移路径是存在的。

## 五、开发路线图

### Phase 1 — MVP 原型（预计 4-6 周）

核心目标：可交互的世界样貌编辑器

- [ ] 项目脚手架（Vite + React + Three.js + R3F）
- [ ] 网格编辑器（创建世界、L1 海洋/大陆绘制）
- [ ] 过渡融合渲染（格线消除、海岸渐变）
- [ ] 2.5D 场景搭建（摄像机倾角 ~60°）
- [ ] L2 自然地形放置（山脉、森林、平原）
- [ ] L3 模型加载与放置（6 个 demo 模型）
- [ ] 点击交互（模型弹出 + 右侧信息面板）
- [ ] 2D/2.5D 视角切换
- [ ] 数据引擎基础架构（单一数据源 + 空间/时间索引模型）
- [ ] IndexedDB 自动持久化
- [ ] JSON 导入/导出功能

### Phase 2 — 多系统数据关联（后续迭代）

- [ ] 人物系统页面与 CRUD
- [ ] 事件系统页面与时间线
- [ ] 体系/元素列表
- [ ] 地图 ↔ 人物 ↔ 事件 ↔ 时间线 数据交叉引用

### Phase 3 — 体验与扩展

- [ ] 模型库扩展（更多预置模型 + 用户导入）

- [ ] 外部软件接入（Obsidian 插件等）
